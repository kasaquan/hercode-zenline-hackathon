=============================================================

NOTE ON THIS DOCUMENT
This file captures the design intent and the current shape of the code. The
authoritative source is the code itself:
  Source/Agents/decision_agent/decision.py   (orchestration + scoring core)
  Source/Agents/decision_agent/analytics.py  (statistical signal analysis layer)
  Source/Agents/decision_agent/pipeline.py   (full Scout + Profiler + Decision run)
Where this doc shows code, it shows current signatures and the key/changed logic

Goal
----
Agent 2 is the main customer-facing decision agent.
  Agent 1 = Scout / Sourcing Agent           (Source/Agents/scout_agent)
  Agent 3 = Company Profile Extractor         (Source/Agents/agent_3.py)
  Agent 2 = Main Decision Agent / Buyer Orchestrator

Core flow:
  Customer query
  → Agent 2 parses request
  → Agent 2 gives product/category keywords to Agent 1
  → Agent 2 gives company context / website / PDF / notes to Agent 3
  → Agent 1 returns out/signals.csv
  → Agent 3 returns out/company_profile.json
  → Agent 2 reads both
  → Agent 2 outputs out/recommendations.csv + out/recommendations.json

(See pipeline.py for the orchestrated version: Agent 2 fans out to Agent 1 and
Agent 3 IN PARALLEL on two threads, joins, then scores — passing their output
in-memory so Decision never re-parses its own inputs.)

Agent 2 answers:
  Given these market signals, and given this company's profile, strategy,
  assortment gaps, innovation appetite, price band, and constraints, what
  should the retailer do next?

Important distinction:
  Agent 1 signal_score = signal-level emerging strength, 0.0–1.0
  Agent 2 final_score  = opportunity-level business decision score, 0–100
Agent 2 uses Agent 1 signal_score mainly for trend_momentum, NOT as the final
ranking score.

Design principle (current): GENERIC-FIRST.
  The agent must work for ANY category (winter jackets, cookware, e-bikes, …),
  not just hard-coded outdoor verticals. Seed expansion and opportunity grouping
  are driven by the customer's product_focus and by each signal's OWN content.
  Outdoor-specific rules survive only as optional enrichment / normalization
  layered on top of the generic logic.

Uses existing SignalRow fields:
  source, market, keyword, signal_name, signal_type, product_name, brand,
  price, rank, url, signal_score, confidence, notes, observed_at,
  artifact_type, artifact_uri, created_by_tool

Input 2: Agent 3 output — out/company_profile.json
Agent 3 schema:

    "primary_cantons": ["ZH", "BE", "VS"],

    "store_count": "integer or range",
    "distribution_model": "direct | retail | hybrid",
    "target_gross_margin": "low (15-25%) | medium (25-40%) | high (40%+)",
    "innovation_appetite": "low | medium | high",
    "supply_chain_constraints": "string",
    "strategic_expansion_focus": "string - what exec strategy says they're moving into",
    "strategic_timeline": "string - when",
    "strategic_rationale": "string - why are they expanding?",
    "strategic_assortment_gaps": "string - INFERRED: what's missing to execute strategy",

    "confidence_by_field": {"field_name": "high | medium | low"},
    "extra": "string - unstructured insights"
  }

Notes on profile handling in decision.py:
- DEFAULT_PROFILE supplies neutral fallbacks and is merged under any loaded
  profile (merge_profile / load_profile). It does not hard-code
  supply_chain_constraints; that field is still read where present and treated
  as absent when missing.
- Agent 3 placeholder prose ("Not available …", "Cannot infer …", "unknown",
  "n/a", …) is treated as ABSENT (see _is_placeholder / _PLACEHOLDER_MARKERS) so
  an honest "I don't know" scores neutral and never pollutes keyword matching.


Expected Outputs
----------------
Output 1: Zenline-compatible recommendation CSV — out/recommendations.csv
Must match RecommendationRow fields (UNCHANGED contract):
  rank, opportunity, first_observed_market, evidence_summary, evidence_urls,
  transferability, coverage_status, recommended_action, confidence, risks

Output 2: Richer JSON for dashboard/debugging — out/recommendations.json
Each recommendation includes:
  rank, opportunity, first_observed_market, evidence_summary, evidence_urls,
  transferability, coverage_status, recommended_action, confidence, risks,
  opportunity_type, signal_markets, scores, missing_evidence, next_step,
  company_alignment_summary, profile_fields_used, source_count, signal_count,
  deduplication_note, analytics_adjustment, analytics

The `scores` object contains:
  evidence_strength, cross_source_validation, trend_momentum,
  swiss_dach_transferability, commercial_potential, sustainability_risk_relevance,
  current_assortment_gap_fit, strategic_gap_fit, company_profile_fit,
  final_score, current_gap_reason, strategic_fit_reason, company_fit_reason,
  weights_used,                 # the dynamic weight vector actually applied
  weight_adjustment_reasons,    # human-readable log of why weights moved
  analytics_adjustment          # multiplier notes from analytics.adjust_final_score


Module: Source/Agents/decision_agent/decision.py
-------------------------------------------------
Imports of note:
  from ..scout_agent.schema import RecommendationRow, write_recommendations
  from . import analytics

Request parsing
  parse_customer_query(query, market="DACH") -> request dict
    Extracts company_name + product_focus from natural-language queries, then
    attaches agent1_seed_keywords via expand_seed_keywords(product_focus).

Seed expansion — GENERIC-FIRST
  expand_seed_keywords(product_focus) -> List[str]
    Always emits product_focus-driven seeds first (work for any category):
      product_focus
      emerging {product_focus}
      {product_focus} trends
      {product_focus} new materials
      {product_focus} rising brands
      {product_focus} competitor assortment
      {product_focus} marketplace bestsellers
      {product_focus} Switzerland DACH
      sustainable {product_focus}
      premium {product_focus}
      lightweight {product_focus}
      repairable {product_focus}
    Category-specific seeds (winter jacket / rain-shell / ultralight / trail /
    climbing) are OPTIONAL additions appended only when the focus clearly matches
    a known vertical — they supplement, never replace, the generic seeds. The old
    generic `else` branch was removed (the generic seeds already cover it).
    Deduplicated, order-preserving.

Opportunity typing — GENERIC
  infer_opportunity_types(row) -> List[str]
    Infers buyer-relevant facets from the signal's OWN text. Returns any of:
      product_type, material, feature, brand, supplier, price_band_gap,
      usage_occasion, merchandising_idea, service_model
    Always returns at least ["product_type"].

Canonical grouping — GENERIC-FIRST
  canonical_opportunity(row, request) -> (label, opportunity_types)
    Grouping key priority (from the signal's own content):
      1. Agent 1 signal_name        (primary key)
      2. product_name
      3. brand/supplier label        (only when brand-led, i.e. no signal_name/
                                      product_name; uses _GENERIC_BRAND_NAMES to
                                      skip "global"/"none"/"unknown"/…)
      4. keyword
      5. "Unspecified opportunity"   (fallback)
    The customer's product_focus is deliberately NOT mixed into the grouping text
    (that would let the query keyword dominate and relabel unrelated signals).
    Optional outdoor normalization: _normalize_outdoor_label(text) collapses
    near-duplicate labels into a shared canonical bucket (winter jackets,
    PFAS-free shells, UV hoodies, ultralight kit, trail-to-city, climbing,
    hydration vests, repairable gear, merino base layers) and merges in extra
    facets. It returns None when nothing matches, so it only refines the generic
    grouping — it is not the main logic.
  group_signals(signals, request) collects rows under each canonical label and
    unions their opportunity_type sets.

Per-opportunity scoring
  score_group(group, profile, request) -> scores dict     # note: takes request
    Computes eight buyer dimensions (each 0–100):
      evidence_strength          = 50*avg_signal + 25*url_coverage + 25*conf_avg
      cross_source_validation    = f(#signal_types, #markets, #sources)
      trend_momentum             = (0.7*avg_signal + 0.3*max_signal)*100
      swiss_dach_transferability = market proximity + use-case/sustainability bumps
      commercial_potential       = competitor/marketplace, price, brand, margin cues
      current_assortment_gap_fit = overlap vs current_product_categories/gaps
      strategic_gap_fit          = overlap vs strategic_expansion_focus/gaps/rationale
      company_profile_fit        = positioning, price band, segments, innovation, margin
    Also computes sustainability_risk_relevance (reported, but NOT part of the
    weighted final_score).

DYNAMIC WEIGHTS (replaces the old fixed final_score formula)
  BASE_WEIGHTS = {                      # baseline contribution, sums to 1.0
    evidence_strength:          0.18,
    cross_source_validation:    0.12,
    trend_momentum:             0.12,
    swiss_dach_transferability: 0.16,
    commercial_potential:       0.10,
    current_assortment_gap_fit: 0.10,
    strategic_gap_fit:          0.12,
    company_profile_fit:        0.10,
  }
  WEIGHT_BUMP = 0.04

  normalize_weights(weights) -> weights        # clamp negatives, rescale to sum 1.0
  get_dynamic_weights(profile, request, rows) -> (normalized_weights, reasons)
    Nudges BASE_WEIGHTS per request, then normalizes. Rules:
      - high innovation_appetite  → +trend_momentum, +strategic_gap_fit
      - low  innovation_appetite  → +evidence_strength, +cross_source_validation
      - high target_gross_margin  → +commercial_potential
      - strategic_expansion_focus or strategic_assortment_gaps present
                                  → +strategic_gap_fit
      - single-source evidence    → +evidence_strength, +cross_source_validation,
                                    −trend_momentum
      - no CH/DACH market evidence→ +swiss_dach_transferability
      - no evidence URLs          → +evidence_strength, −trend_momentum
      - query intent mentions trend/emerging/rising/new → +trend_momentum
      - query intent mentions premium/margin/profit     → +commercial_potential
    Every applied rule appends a line to `weight_adjustment_reasons`.
  compute_final_score(scores, weights) -> float    # weighted sum of the 8 dims

  score_group returns the dimension scores PLUS weights_used and
  weight_adjustment_reasons so the JSON is fully auditable.

Decision logic (unchanged in spirit)
  choose_action(...)      → Launch / Test / Buy / Contact supplier/brand /
                            Monitor / Reposition existing assortment / Ignore
  confidence_label(...)   → low … high
  transferability_text(...), risks_and_missing(...), summarize_evidence(...),
  next_step_text(...), infer_coverage_status(...), infer_first_observed_market(...)

Assembly
  build_recommendations(signals, profile, request) -> result dict
    1. group_signals
    2. analytics_results = analytics.analyze_signals(signals, use_embeddings=False)
       (run ONCE over all signals)
    3. for each opportunity:
         scores = score_group(group, profile, request)
         adjusted, notes = analytics.adjust_final_score(scores["final_score"],
                                                        analytics_results, opportunity)
         scores["final_score"] = adjusted          # analytics multiplier applied
         scores["analytics_adjustment"] = notes
         choose_action / confidence / risks → rich rec (incl. an `analytics`
         block summarizing trend, saturation, diversity, source quality,
         anomalies, geography, clustering)
    4. sort by scores.final_score desc, assign rank
    Returns: recommendations, methodology_note, known_limitations, input_summary

I/O
  read_signals / signals_from_rows  → normalize Agent 1 rows (file or in-memory)
  load_profile / merge_profile      → Agent 3 profile over DEFAULT_PROFILE
  to_recommendation_rows + write_recommendations → out/recommendations.csv
  write_json                        → out/recommendations.json


Module: Source/Agents/decision_agent/analytics.py
--------------------------------------------------
Adds deterministic data-science rigor on top of the scoring core. Requires numpy.
Public surface used by decision.py:

  analyze_signals(signals, use_embeddings=False, verbose=False) -> dict
    Runs all analyses once and returns:
      clustering      — keyword/Jaccard clustering (or sentence-transformers +
                        KMeans when use_embeddings=True and libs are installed;
                        falls back to keyword clustering otherwise)
      trend           — trend velocity / acceleration from observed_at timestamps
      saturation      — brand diversity / market crowding (saturation_level)
      anomalies       — unusual signal scores and price points (IQR-based)
      geography       — CH vs DACH vs global concentration
      source_quality  — source credibility weighting (quality_score)
      diversity       — keyword/brand/source diversity (diversity_score)

  adjust_final_score(base_score, analytics, opportunity_name)
        -> (adjusted_score, adjustment_notes)
    Multiplies base_score by signals from the analytics bundle:
      trend accelerating +15% / decelerating −15%
      saturation high −10% / low +10%
      diversity >0.7 +5% / <0.3 −10%
      source quality >0.8 +5% / <0.5 −15%
      geography ch_focused or dach_focused +8%
    adjustment_notes is a {factor: explanation} map surfaced in the JSON as
    scores.analytics_adjustment.

(Compatibility) analytics.py is kept Python-3.7 friendly — list comprehensions
are used instead of the walrus operator.

README / SUBMISSION Text
------------------------
Agent 2 is the main customer-facing Assortment Decision Agent. It parses a
retailer request, converts the product/category need into GENERIC, product-focus
driven seed keywords for Agent 1, waits for Agent 1's Signal Rows and Agent 3's
company profile, then produces ranked retail recommendations.

Agent 1 provides signal-level emerging evidence. Agent 2 groups and deduplicates
related signals into canonical opportunities (keyed generically on each signal's
own content, with optional category normalization) and uses Agent 1 signal_score
mainly as trend_momentum.

Agent 3 provides company strategy and assortment context: current product
categories, current assortment gaps, strategic expansion focus, strategic
assortment gaps, price positioning, gross margin target, innovation appetite, and
supply chain constraints.

Agent 2 combines these into opportunity-level scoring across eight buyer
dimensions (evidence strength, cross-source validation, trend momentum,
Swiss/DACH transferability, commercial potential, current assortment gap fit,
strategic gap fit, company profile fit). The dimension WEIGHTS are dynamic:
they adapt to customer query intent, the Agent 3 profile, and Agent 1 evidence
quality, and every adjustment is logged. A deterministic analytics layer
(clustering, trend velocity, saturation, anomalies, geography, source quality,
diversity) then applies a transparent multiplier to the final score.

Output is written to out/recommendations.csv in the Zenline Recommendation Row
contract and to out/recommendations.json with scoring (incl. weights_used and
weight_adjustment_reasons), analytics, risks, missing evidence, next steps, and
company alignment details.


Summary
-------
  Agent 1 finds signals.
  Agent 3 explains the company.
  Agent 2 decides what the company should do — generically across categories,
  with dynamic weights and a statistical analytics layer.

  out/recommendations.csv  = jury-facing Zenline contract
  out/recommendations.json = dashboard/debug/scoring details
